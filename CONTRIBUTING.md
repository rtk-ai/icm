workspace with CLI, MCP, storage, and integration code.

## What is ICM?

ICM (Infinite Context Memory) gives AI agents persistent memory across tools and
sessions. It stores memories, transcripts, and knowledge graph concepts in a
local database, exposes them through CLI and MCP tools, and installs integrations
for agent environments such as Claude Code, Codex, Gemini, Cursor, Copilot, Amp,
and others.

The project is pre-1.0 and under active development. APIs, hooks, and MCP
configuration details can change between minor releases.

## Design Philosophy

### Local First

ICM's default behavior is local-first: one binary, local storage, no required
hosted service. Contributions should preserve that model unless the feature is
explicitly optional.

### One Memory, Many Agents

All supported clients should be able to read and write the same memory corpus.
When adding integrations, keep behavior consistent across CLI, MCP, hooks, and
generated instruction files where applicable.

### Trustworthy Recall

Memory quality matters more than volume. Prefer precise storage, deduplication,
topic hygiene, and clear retrieval behavior over broad noisy extraction.

### Safe Automation

ICM can edit agent config files and install hooks. Any code that touches user
files, shell commands, paths, credentials, or config must be conservative,
reversible where possible, and covered by tests.

### Minimal Runtime Burden

ICM should stay fast and practical for daily agent workflows. Avoid unnecessary
startup work, blocking network calls in local paths, and large dependency
additions without a strong reason.

## Project Structure

| Path | Purpose |
|------|---------|
| `crates/icm-cli` | CLI, init/install/uninstall flows, hooks, dashboard, HTTP API |
| `crates/icm-mcp` | MCP JSON-RPC server and tool surface |
| `crates/icm-store` | Storage backends, memory search, transcripts, memoirs |
| `crates/icm-core` | Shared domain types and core logic |
| `docs/` | User, product, architecture, backend, and integration docs |
| `.github/workflows/` | CI, release, PR target checks |
| `scripts/` | Benchmarks and quality scripts |

Follow existing folder boundaries. Do not reorganize unrelated code as part of a
feature PR.

## Commit Messages & Changelog

ICM uses [Conventional Commits](https://www.conventionalcommits.org/) and
[release-please](https://github.com/googleapis/release-please) to generate
release notes and version bumps. Do not edit `CHANGELOG.md` manually if one is
generated by release automation.

### Commit format

```text
<type>(<scope>): <short description>
```

| Type | Semver Impact | When to Use |
|------|---------------|-------------|
| `feat` | Minor | New features, new tools, new integrations |
| `fix` | Patch | Bug fixes and corrections |
| `perf` | Patch | Performance improvements |
| `refactor` | - | Code restructuring without behavior changes |
| `docs` | - | Documentation only |
| `chore` | - | Maintenance, CI, dependencies |
| `feat!` / `fix!` | Major | Breaking changes |

Use scopes such as `cli`, `mcp`, `store`, `hook`, `init`, `dashboard`, `docs`,
`ci`, or a backend/integration name.

Examples:

```text
feat(mcp): add transcript search tool
fix(init): preserve existing Codex MCP settings
perf(store): cache topic health calculations
docs(integrations): clarify OpenCode plugin setup
```

## Branch Naming Convention

Use a prefix that matches the change type, followed by a short kebab-case
description.

| Prefix | When to Use |
|--------|-------------|
| `fix/` | Bug fixes, corrections, minor adjustments |
| `feat/` | New features, backends, tools, integrations |
| `docs/` | Documentation-only changes |
| `chore/` | CI/CD, dependencies, maintenance |

Examples:

```text
fix/codex-hook-extraction-throttle
feat/http-mcp-proxy
docs/contributing-security
chore/update-release-workflow
```

## Pull Request Process

### Scope Rules

Each PR must focus on a single feature, fix, or documentation change.
Out-of-scope refactors, formatting churn, or unrelated cleanup should go in a
separate PR.

For large features or refactors, prefer multiple reviewable PRs over one large
diff. Each part should compile, test, and be useful on its own.

### 1. Create Your Branch

Contributor PRs must target the `develop` branch. The `main` branch is reserved
for stable releases and maintainer release automation.

```bash
git checkout develop
git pull origin develop
git checkout -b feat/your-clear-description
```

### 2. Make Your Changes

- Keep changes within the relevant crate, docs section, or workflow.
- Match existing Rust style and error handling patterns.
- Prefer existing helper APIs over new abstractions.
- Do not add dependencies unless the benefit clearly outweighs supply-chain and
  binary-size cost.
- Avoid obvious comments; comments should explain invariants, constraints, or
  non-obvious design choices.

### 3. Add Tests

Every behavior change should include tests. Documentation-only changes generally
do not need tests, but commands and examples should be accurate.

### 4. Update Documentation

Documentation updates are required for user-facing behavior, new flags, new MCP
tools, integration changes, setup changes, and backend changes.

### 5. Open the PR Against `develop`

Open your PR with base branch `develop`. The repository has a PR target check
that flags contributor PRs opened against `main`.

### 6. Review Process

1. Maintainers review code quality, design fit, and project scope.
2. CI runs formatting, Clippy, tests, and security checks.
3. Address review comments and CI failures.
4. Maintainers merge when the PR is ready.

### 7. Integration & Release

Merged changes land on `develop` first. Stable releases are cut from `main` by
maintainers through the release workflow.

```text
your branch -> develop (review + CI + integration) -> main (stable release)
```

## Testing

ICM is a Rust workspace. Run the smallest useful test first while developing,
then run the broader gates before opening a PR.

### Common Commands

| Purpose | Command |
|---------|---------|
| Format check | `cargo fmt --all -- --check` |
| Lint | `cargo clippy --workspace --all-targets -- -D warnings` |
| Full test suite | `cargo test --workspace` |
| Test one crate | `cargo test -p icm-cli` |
| Build CLI | `cargo build -p icm-cli` |
| Build release CLI | `cargo build --release -p icm-cli` |

### Pre-Commit Gate

Run these before opening a PR:

```bash
cargo fmt --all -- --check
cargo clippy --workspace --all-targets -- -D warnings
cargo test --workspace
```

### PR Testing Checklist

- [ ] Unit or integration tests added/updated for behavior changes
- [ ] CLI examples manually checked when flags or command output changed
- [ ] MCP JSON-RPC behavior checked when tool contracts changed
- [ ] Config-writing paths tested with temp directories when init/uninstall logic changed
- [ ] New dependencies justified and reviewed
- [ ] `cargo fmt --all -- --check` passes
- [ ] `cargo clippy --workspace --all-targets -- -D warnings` passes
- [ ] `cargo test --workspace` passes

## Documentation

|--------------|-------------------|
| CLI flags or user workflow | `README.md`, `docs/guide.md` |
| Agent integration setup | `README.md`, `docs/integrations.md` |
| Architecture or crate boundaries | `docs/architecture.md` |

| MCP tools or contracts | `README.md`, `docs/integrations.md` |

explanations.
